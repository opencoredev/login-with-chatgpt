---
title: Experimental ChatGPT voice transport
description: An unsupported private transport for GPT Live audio, with interruption, transcripts, captions, and explicit product limitations.
---

The SDK can experimentally open the signed-in user's private ChatGPT voice
transport over WebRTC. Audio flows directly between the browser and the voice
peer after a one-time SDP
exchange through your backend. Access, refresh, and ChatGPT web-session
credentials never enter the WebRTC/browser response.

<Callout type="warn" title="Unsupported private ChatGPT transport">
  This integration targets ChatGPT's subscription-backed web transport. Its
  endpoints and data-channel schema are not a documented public OpenAI API and
  can change or stop working without notice. This is not an official capability
  of the OpenAI SDK, and a ChatGPT subscription is not an API credential. If
  your product uses an OpenAI API key, prefer the
  official [`/v1/realtime/calls` WebRTC interface](https://developers.openai.com/api/docs/guides/realtime-webrtc)
  instead.
</Callout>

<Callout type="warn" title="Direct /wm sessions are voice-only">
  `/wm` accepts only ChatGPT's reserved device-tool IDs (`timer`, `stopwatch`,
  `precise_location`, and `led`) in `client_tools`; arbitrary application
  function schemas are rejected. The desktop app's thread-scoped Realtime
  mediation is a separate app-server protocol; its handoff items are not
  emitted to a direct browser `/wm` data channel. Do not turn caption or
  transcript events into tool calls. Use the server package's app-server bridge
  below when you intentionally accept that private desktop protocol, or use a
  supported tool-capable Realtime API.
</Callout>

OpenAI currently documents three ChatGPT product options: **Live** (GPT-Live-1
on paid plans), **Advanced** (the previous realtime experience), and **Standard**
(turn-by-turn transcription). Product availability depends on plan, region,
device, and rollout. The private route names below are observed implementation
details; they are not a documented contract mapping those product options. See
OpenAI's current [ChatGPT Voice documentation](https://help.openai.com/en/articles/20001274/)
for the authoritative product capability and availability matrix.

## Native tools through app-server mediation

`@opencoredev/loginwithchatgpt-server` exports
`ChatGPTRealtimeAppServerSession` for the desktop-style path. It preserves the
native WebRTC audio session and interruption behavior while app-server receives
native handoffs, runs allowlisted application tools, and appends a concise
spoken result. It does not infer actions from transcripts and does not introduce
a second model.

Run this class only on your backend. The `codex` executable must be installed
and available on `PATH` (or supplied with `command`), and ChatGPT tokens must
come from your encrypted, user-bound server session:

```ts title="realtime-tools.ts"
import {
  ChatGPTRealtimeAppServerSession,
  type RealtimeDynamicTool,
} from "@opencoredev/loginwithchatgpt-server";

const tools: RealtimeDynamicTool[] = [{
  type: "function",
  name: "email_list_threads",
  description: "List recent email threads for the authenticated app user.",
  inputSchema: {
    type: "object",
    properties: { count: { type: "integer", minimum: 1, maximum: 20 } },
    required: ["count"],
    additionalProperties: false,
  },
}];

const live = new ChatGPTRealtimeAppServerSession({
  tokens: await loadServerOnlyChatGPTTokens(appUser.id),
  refreshTokens: () => refreshServerOnlyChatGPTTokens(appUser.id),
  tools,
  executeTool: async ({ callId, name, arguments: args }) => ({
    output: await executeAllowlistedTool(appUser, callId, name, args),
  }),
});

live.onEvent((event) => publishToAuthenticatedStatusStream(appUser.id, event));

const answerSdp = await live.start({
  sdp: browserOfferSdp,
  voice: "vale",
  // Propagate the model selected by this signed-in user.
  model: "gpt-5.6-luna",
});
```

`start()` returns the WebRTC answer SDP. Keep the session object in a
user-scoped server store, return only an opaque session id plus SDP to the
browser, and call `close()` when the call ends. Propagating the selected model
is important: otherwise a delegated turn can inherit a different
model-specific entitlement.

Tool results are JSON-serializable `output`. For consequential actions, return
`pendingConfirmation` and render its safe `review` payload in your UI:

```ts
return {
  output: { status: "pending_confirmation", draftId },
  pendingConfirmation: {
    review: { draftId, to, cc, bcc, subject, body },
  },
};
```

The bridge immediately resolves the app-server request with that pending
result, keeping native voice responsive. The action remains application-owned
until an explicit UI action calls `resolveConfirmation(callId, confirmation)`.
Implement `confirmTool` to re-read the resource, verify the authenticated user,
and perform the final action. Speech such as “send it” may request this review,
but must never substitute for application confirmation.

Bridge events are `session.started`, `session.closed`, `handoff`,
`tool.running`, `tool.completed`, `tool.pending_confirmation`, `tool.failed`,
and `error`. Never put bearer tokens, raw private tool results, or credentials
in the browser event stream.

## Start a call

GPT Live `/realtime/wm` does not accept the Codex device-login token used by the
rest of this SDK. Configure the handler with a server-only auth callback backed
by an encrypted ChatGPT web session. The callback should also verify that the
ChatGPT email matches your application's authenticated user:

```ts title="server.ts"
import {
  exchangeChatGPTRealtimeWebSession,
  resolveConfig,
} from "@opencoredev/loginwithchatgpt-core";
import { createChatGPTHandler } from "@opencoredev/loginwithchatgpt-server";

const config = resolveConfig();
const auth = createChatGPTHandler({
  secret: process.env.LWC_SECRET,
  realtime: {
    async getAuth({ request }) {
      const appUser = await requireAppUser(request);
      const encrypted = await loadUserRealtimeSession(appUser.id);
      const stored = await decrypt(encrypted);
      const live = await exchangeChatGPTRealtimeWebSession({
        config,
        sessionToken: stored.sessionToken,
        deviceId: stored.deviceId,
        sessionCookies: stored.sessionCookies,
        onSessionUpdate: async (next) => {
          await saveUserRealtimeSession(appUser.id, await encrypt({ ...stored, ...next }));
        },
      });
      if (live.email.toLowerCase() !== appUser.email.toLowerCase()) {
        throw new Error("ChatGPT Live identity does not match the app user");
      }
      return live;
    },
  },
});
```

Then connect from a user gesture so the browser can grant microphone access and
play audio:

```bash
bun add @opencoredev/loginwithchatgpt-core
```

```ts title="voice.ts"
import { connectChatGPTRealtime } from "@opencoredev/loginwithchatgpt-core";

const call = await connectChatGPTRealtime({
  session: {
    voice: "juniper",
    transport: "wm",
    language: "en-US",
  },
  onStateChange: (state) => console.log(state),
  onEvent: (event) => {
    if (event.type === "user_transcription_text") console.log(event);
  },
});

// Explicit controls are useful for buttons and push-to-talk.
call.stopSpeaking();
call.setInputMuted(true);
call.setOutputMuted(false);

// Always release the microphone, peer, data channel, and audio graph.
call.close();
```

`connectChatGPTRealtime()` does the following:

1. captures audio with echo cancellation, noise suppression, and automatic gain;
2. creates a `max-bundle` WebRTC peer and negotiated data channel id `0`;
3. posts the local SDP and session options to `/api/chatgpt/realtime`;
4. applies the SDP answer, plays the remote audio track, and decodes events;
5. monitors local voice activity while the model speaks and sends
   `stop_speaking` for natural barge-in.

Pass `bargeIn: false` to disable local interruption detection, or tune it:

```ts
bargeIn: { threshold: 0.045, holdMs: 120, cooldownMs: 800 }
```

You may supply `mediaStream`, `mediaConstraints`, `peerConnection`, or
`audioElement` when your application owns those objects. A supplied stream is
not stopped by `close()`; a stream created by the SDK is.

## Connection API

The resolved `ChatGPTRealtimeConnection` exposes:

| Member | Purpose |
| --- | --- |
| `state` | Current `connecting`, `listening`, `thinking`, `speaking`, or other reported state. |
| `peerConnection`, `dataChannel`, `mediaStream`, `audioElement` | Escape hatches for media/UI integration. |
| `send(event)` | Send a forward-compatible raw protocol event. |
| `action(name, payload?)` | Send an `action_request`. |
| `startListening()`, `stopListening()`, `resumeListening()` | Listening controls. |
| `stopSpeaking()` | Interrupt current model speech. |
| `relayMessage(text, metadata?)` | Inject a typed user message into GPT Live. |
| `setInputMuted(boolean)`, `setOutputMuted(boolean)` | Local media controls. |
| `close()` | Idempotently release all SDK-owned resources. |

For push-to-talk, disable automatic barge-in, call `startListening()` on press,
and `stopListening()` on release.

GPT Live currently does not support video or screen sharing. The browser helper
rejects video constraints and streams with video tracks. Its peer connection
still creates an inert send-only video transceiver because the observed private
handshake expects that SDP m-line; this is transport compatibility, not a video
feature. OpenAI documents video and screen sharing only for eligible subscribers
using Advanced in supported mobile apps, not as a capability of this SDK.

## Session schema

The browser sends ergonomic camel-case `ChatGPTRealtimeSessionOptions`. The
core package converts them into the upstream wire keys shown below.

| SDK option | Upstream key | Default / behavior |
| --- | --- | --- |
| `transport` | endpoint selection | `"wm"` (GPT Live); `"vp"` and `"vps"` are explicit compatibility options |
| `voice` | `voice` | `"juniper"` |
| `voiceMode` | `voice_mode` | `"wingman"` for `/wm` |
| `model` | `requested_default_model`, `model_slug` | Empty for `/wm`, allowing the subscription to select GPT Live; explicitly required for compatibility paths. |
| `advancedModel` | `model_slug_advanced` | Private `vp` compatibility field; no product-feature guarantee. |
| `language` | `language_code` | `null`; browser helper may pass locale |
| `conversationId` | `conversation_id` | `null` for a new conversation |
| `parentMessageId` | `parent_message_id` | omitted |
| `timezone`, `timezoneOffsetMinutes` | `timezone`, `timezone_offset_min` | browser timezone; server fallback is UTC / `0` |
| `clientTools` | `client_tools` | Must be empty. `/wm` rejects arbitrary application IDs. |
| `conversationMode` | `conversation_mode` | `{ "kind": "primary_assistant" }` |
| `historyAndTrainingDisabled` | `history_and_training_disabled` | `false` for `/wm`; OpenAI currently denies `/wm` sessions when true |
| `enableMessageStreaming` | `enable_message_streaming` | `true` by default |

The fixed `/wm` wire payload includes `backend_reasoning_effort: "high"`,
`chat_mode: "chat"`, and an empty `client_tools` array, matching the current
ChatGPT clients.
The SDK deliberately does not expose arbitrary private metadata, prompt
overrides, delegation controls, or model-tool fields as supported features.

`voice_session_id` and `voice_status_request_id` are generated per call. Use
`buildChatGPTRealtimeSession()` when you need the exact wire object without
opening a connection.

## HTTP endpoint

`POST /api/chatgpt/realtime` is cookie-authenticated and protected by the same
origin checks as `/responses`.

```json title="request"
{
  "sdp": "v=0\r\n…",
  "session": {
    "voice": "juniper",
    "transport": "wm",
    "language": "en-US"
  }
}
```

A successful response is `201 application/sdp` containing the WebRTC answer.
Errors are JSON:

| Status | Error |
| --- | --- |
| `400` | `invalid_realtime_request` |
| `401` | `not_authenticated` |
| `403` | `origin_not_allowed`, `realtime_mode_not_allowed`, or `realtime_transport_not_allowed` |
| `413` | `realtime_request_too_large` |
| `501` | `realtime_web_auth_required` when `/wm` has no server auth callback |
| upstream / `502` | `realtime_request_failed` |

Configure the route with:

```ts
createChatGPTHandler({
  enableRealtime: true,
  realtime: {
    maxRequestBytes: 256 * 1024,
    allowedTransports: ["wm"],
    allowedModes: ["wingman"],
    sessionDefaults: { transport: "wm", historyAndTrainingDisabled: false },
    getAuth: async ({ request }) => getUserBoundWebAuth(request),
  },
});
```

The handler converts the request to multipart `sdp` + `session` fields and
posts to the selected ChatGPT endpoint:

| Transport | Upstream signaling endpoint |
| --- | --- |
| `wm` | `POST https://chatgpt.com/realtime/wm` |
| `vp` | `POST https://chatgpt.com/realtime/vp?dcid=0` |
| `vps` | `POST https://chatgpt.com/realtime/vps?dcid=0` |

`wm` is the default because it is the currently observed GPT Live transport.
The server permits only `wm` unless `realtime.allowedTransports` explicitly
opts into a compatibility path. `vp` and `vps` are undocumented, may disappear,
and do not make Advanced or Standard product features available through this
SDK. Codex OAuth tokens do not authorize `wm`; `getAuth` must return a
short-lived token minted for the ChatGPT web client.
`exchangeChatGPTRealtimeWebSession()` performs
that server-only exchange and validates the token client id.

## Data-channel schema

Messages arrive in a nested
`{ "type": "data_message", "data": "<json>" }` envelope. The parser accepts
strings, `ArrayBuffer`, typed arrays, byte arrays, and numeric-key byte objects.
Unknown event types pass through unchanged. Event-specific fields may be on the
event or its `payload`; `getChatGPTRealtimePayload(event)` normalizes both forms.

Client events must use that same outer envelope. `/wm` silently ignores a raw
inner action or relay event. `encodeChatGPTRealtimeEvent()` and every connection
method add the required framing.

Observed event names exported in `CHATGPT_REALTIME_EVENT_TYPES`:

| Area | Event types |
| --- | --- |
| State/control | `state_update`, `action_request`, `goodbye`, `close_request`, `close_ready`, `track_state`, `speaking_update` |
| Conversation/streaming | `conversation_update`, `streaming_message_update`, `full_chat_message`, `chat_message_delta`, `relay_message`, `relay_message_processed` |
| Speech/text | `user_transcription_text`, `live_captioning_text` |
| Reserved client tools | `client_tool_invoke`, `client_tool_update`, `client_tool_result` |

This catalog is intentionally narrow. The parser passes unknown events through
for forward compatibility, but their presence is not an SDK capability claim.

The client-tool parsing and result helpers remain available for compatibility
with ChatGPT's reserved first-party tools, but the SDK does not advertise
arbitrary schemas in `/wm`.

Control messages use this shape:

```json
{
  "type": "action_request",
  "payload": { "action": "stop_speaking" }
}
```

Known actions are `start_listening`, `stop_listening`, `stop_speaking`,
`resume_listening`, and `relay_message`. A conversational text relay is not an
`action_request`; `createChatGPTRealtimeRelayMessage()` builds the currently
observed private message shape. It is not a function call or tool result. Use
`createChatGPTRealtimeAction()`, `encodeChatGPTRealtimeEvent()`, and
`parseChatGPTRealtimeEvent()` for a custom transport.

## Low-level server use

Custom backends can call `createChatGPTRealtimeCall()` directly with a resolved
config, an auth callback, SDP, and session options. This is a server-only
primitive because `getAuth()` returns bearer material. Some runtime HTTP stacks
may be challenged by ChatGPT's web edge; inject a compatible server-side
`fetch` through `resolveConfig({ fetch })` when necessary. Never solve that by
moving the ChatGPT access token into the browser.

The web-session credential is longer-lived and more sensitive than the minted
access token. Encrypt it at rest, bind it to the application's user identity,
never log it, and never put it in local storage or a client-readable cookie.
ChatGPT's web edge also requires a stable `oai-did` device identity and can
rotate its NextAuth session into numbered cookie chunks. Persist the
`deviceId` and `sessionCookies` delivered to `onSessionUpdate`; serialize
concurrent exchanges for one user so two requests cannot race that rotation.
The helper distinguishes a true 401 expiry from a retryable 403 edge rejection.

Treat every client tool invocation as untrusted input. Revalidate it against a
server-side allowlist, bind execution to the authenticated application user,
make calls idempotent by call id, and return only structured results. Keep
drafting separate from sending or other consequential mutations, and require an
explicit application UI confirmation immediately before the mutation.
